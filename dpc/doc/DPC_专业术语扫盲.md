# DPC 坏点校正 —— 专业术语扫盲手册

> 本文档面向零基础同学，用通俗语言解释 DPC 项目涉及的所有专业术语。
> 阅读后你应该能看懂设计文档里的每一个名词在说什么。

---

## 一、图像与相机基础

### 1.1 像素 (Pixel)

图像的最小单位。一张 1920×1080 的照片，就是 1920 列 × 1080 行 = 约 207 万个像素排列成的矩形网格。每个像素存储一个颜色值。

**打个比方**：像素就像马赛克拼图里的一小块瓷砖，无数小块拼在一起就是一幅完整的画。

---

### 1.2 RAW 格式

相机传感器拍到的"生肉"数据。没有经过任何加工处理，每个像素只记录一种颜色的亮度（红/绿/蓝其中之一）。

| 格式    | 每像素位数 | 亮度等级     | 比喻               |
| ----- | ----- | -------- | ---------------- |
| RAW8  | 8 bit | 0~255    | 256 级灰度          |
| RAW10 | 10 bit | 0~1023   | 1024 级灰度（更细腻）    |
| RAW12 | 12 bit | 0~4095   | 4096 级灰度（专业相机级别） |

**为什么叫 RAW？** 因为这就像菜市场买的"生肉"，需要经过一系列烹饪（ISP 处理）才能变成一道能上桌的菜（最终图像）。坏点校正就是烹饪过程中的一步。

---

### 1.3 Bayer 阵列（拜耳阵列）

相机传感器上覆盖了一层彩色滤光片，排列方式像棋盘格：

```
R  G  R  G  R  G  ...    R = 红色滤镜
G  B  G  B  G  B  ...    G = 绿色滤镜（人眼对绿色最敏感，所以 G 最多）
R  G  R  G  R  G  ...    B = 蓝色滤镜
G  B  G  B  G  B  ...
```

**关键点**：每个像素只能感受一种颜色！比如一个"R"位置的像素只知道红色有多亮，不知道绿色和蓝色。后续需要通过"去马赛克"算法把缺失的颜色补全。

**对我们的影响**：在 Bayer 阵列中，同一个颜色的像素**间隔为 2**。比如相邻两个 R 像素之间隔着 1 个 G 像素，所以校正时找"同色邻域"要跳 2 步。

---

### 1.4 坏点 (Defect Pixel / Bad Pixel)

相机传感器上某些像素"坏掉了"——永远输出错误的值。常见类型：

| 坏点类型 | 表现                 | 比喻          |
| ---- | ------------------ | ----------- |
| 亮点   | 始终输出最大亮度（全白/全亮）    | 一直亮着的坏灯泡    |
| 暗点   | 始终输出最小亮度（全黑）       | 永远不亮的灯泡     |
| 漂移点  | 输出值偏离正常值一个固定偏差     | 指针歪了的天平     |
| 随机坏点 | 输出值不稳定，每次都不一样      | 接触不良的灯      |

**坏点校正的目的**：用周围正常像素的值来"猜"这个坏点本该是什么值，就像你看到一张照片上有个黑点，用周围的颜色把它涂掉。

---

### 1.5 邻域 (Neighborhood)

某个像素周围的一圈像素。比如 3×3 邻域就是以该像素为中心的 3 行 × 3 列小窗口：

```
  .  .  .
  . [X] .     ← [X] 是当前像素，周围 8 个 . 就是它的邻域
  .  .  .
```

---

### 1.6 邻域平均与线性插值

**邻域平均**：把周围正常像素的值加起来取平均 → 用平均值替换坏点。

```
比如左邻域=100, 右邻域=120, 上邻域=110
坏点替换值 = (100 + 120 + 110) / 3 = 110
```

**线性插值**：根据左右（或上下）两个正常像素的值，线性推算出中间的值。

```
左值=100, 右值=120
中间值 = (100 + 120) / 2 = 110
```

**在像素间距为 2 的 Bayer 阵列中，两种方法结果相同。**

## 二、FPGA 硬件基础

### 2.1 FPGA 是什么

Field Programmable Gate Array，**现场可编程门阵列**。

**通俗理解**：FPGA 是一块"万能芯片"。普通芯片（比如你电脑的 CPU）出厂后功能就固定了；FPGA 可以让你**用代码"画"出自己想要的电路**。你用 Verilog 描述想要什么电路，工具就把 FPGA 内部的逻辑单元连接成那个电路。

**比喻**：普通芯片是建好的毛坯房，FPGA 是一盒乐高积木，你想搭什么就搭什么。

---

### 2.2 时钟 (Clock) 与时钟频率

数字电路的心脏——一个不停振荡的方波信号，所有操作都按它的节拍进行。

```
时钟信号：
    ┌─┐   ┌─┐   ┌─┐   ┌─┐
    │ │   │ │   │ │   │ │
────┘ └───┘ └───┘ └───┘ └──  ...

每个上升沿（┌─ 那个跳变）做一次操作

125 MHz = 每秒 1.25 亿个上升沿 = 每 8 纳秒做一次操作
```

**时钟频率越高** → 处理越快，但电路设计难度越大（信号来不及稳定就要被采样）。

---

### 2.3 时钟域 (Clock Domain)

同一个时钟信号控制的电路区域。不同时钟域之间传递数据需要特殊处理。

| 本项目时钟 | 频率      | 用途          |
| ------ | ------- | ----------- |
| sys_clk | 125 MHz | 像素处理        |
| ddr_clk | 1500 MHz | 内存读写（本项目不涉及） |

**为什么需要多个时钟域？** 就像工厂的不同流水线以不同速度运转——像素处理每个时钟处理 1 个像素就够了，但内存读写需要高速运转才能跟上数据量。

---

### 2.4 LUT (查找表 / Look-Up Table)

FPGA 最基本的逻辑单元。一个 LUT 就像一个"真值表"——你告诉它，输入是什么组合时输出 0 还是 1。

**比喻**：LUT 就像一个小型"如果-那么"表，FPGA 里有几十万个这样的小表，组合起来实现任意逻辑。

---

### 2.5 FF (触发器 / Flip-Flop)

存储 1 个 bit（0 或 1）的元件。每个时钟上升沿到来时，把输入值锁存并输出。

**比喻**：FF 就像一个只能装 1 个比特的小抽屉，每当时钟敲一下，就打开抽屉换一个新比特进去。

---

### 2.6 BRAM (Block RAM，块状存储器)

FPGA 内部自带的**批量存储器**，可以存几千到几万个比特。

| 对比   | LUT/FF                 | BRAM            |
| ---- | ---------------------- | --------------- |
| 用途   | 做逻辑运算                  | 存大量数据           |
| 容量   | 极小（1 bit ~ 几十 bit）      | 大（18Kb ~ 36Kb） |
| 数量   | 几万 ~ 几十万个              | 几百 ~ 几千个        |
| 本项目约束 | 每个 IP ≤ 总计的 5%          | 每个 IP ≤ 2%（本项目要求 0 块） |

**为什么 DPC 要求 0 块 BRAM？** 因为我们的行缓冲数据量不大，可以用 LUT + FF 拼出来的"分布式 RAM"代替。合理分派资源。

---

### 2.7 分布式 RAM (Distributed RAM)

用 LUT + FF 拼出来的存储器。不像 BRAM 那样是现成的，而是自己"手工打造"。容量小但灵活。

**比喻**：BRAM 是买来的成品书架；分布式 RAM 是用木板自己钉的小格子。

---

### 2.8 DSP (数字信号处理器)

FPGA 内嵌的专用乘法/加法单元，做数学运算特别快。

**本项目基本不涉及 DSP**，因为坏点校正的核心运算是加法、比较、移位，不需要乘法。

---

### 2.9 流水线 (Pipeline)

把一个复杂操作拆成多个简单步骤，每个时钟周期完成一步。就像工厂流水线：

```
没有流水线：一个工人完成所有工序 → 每 10 分钟产出一个
有流水线：10 个工位各做一步 → 每 1 分钟产出一个（第一个要等 10 分钟）

我们的设计：
Stage1 → Stage2 → Stage3 → Stage4 （4 级流水线）
每个时钟都有新数据进入，每个时钟都有结果输出。
```

**流水线延迟**：数据从进入到出来经过的时钟数。本项目约 4~5 拍 → 即第 N 个像素在第 N+5 个时钟才输出，但每个时钟都有输出。

**吞吐量**：本项目 1 像素/时钟 → 每个时钟周期处理 1 个像素，实时处理，不卡顿。

---

### 2.10 关键路径 (Critical Path)

电路中耗时最长的那条信号通路。它决定了整个电路能跑多快。

**比喻**：一条流水线能跑多快，取决于最慢的那个工位。

---

### 2.11 时序收敛 (Timing Closure)

综合工具检查后确认：在所有条件下，信号都能在时钟周期内稳定到达目的地。

**如果时序不收敛**：电路可能出错——信号还没稳定就被下一个时钟采走了，就像接力赛中还没拿到棒子就被迫起跑。

---

## 三、AXI-Stream 视频接口

### 3.1 什么是 AXI-Stream

ARM 公司制定的一种**流式数据传输协议**，专门用于持续不断的数据传输（比如视频流）。

**核心概念**：一条单向"水管"，数据像水流一样源源不断地流过。

---

### 3.2 接口信号详解

信号名中的前缀 `s_axis_` 表示**输入端**（Slave），`m_axis_` 表示**输出端**（Master）。

| 信号名 | 方向 | 通俗解释 |
|--------|------|----------|
| `tvalid` | 发送→接收 | **"我这数据是有效的，请收"**。发送方举起的手。 |
| `tready` | 接收→发送 | **"我准备好了，发过来吧"**。接收方举起的手。 |
| `tdata` | 发送→接收 | **数据本身**——像素的颜色值。 |
| `tuser` | 发送→接收 | **"这是新一帧的第一个像素"**（SOF，Start of Frame） |
| `tlast` | 发送→接收 | **"这是当前行的最后一个像素"**（EOL，End of Line） |
| `tkeep` | 发送→接收 | 指示 tdata 中哪些字节有效（本项目较少使用） |

**握手规则**：只有当 `tvalid=1` **且** `tready=1` 的同一个时钟沿，数据才真正传输。

**比喻**：
- tvalid="我手上拿着货"
- tready="我有空位放货"
- 两人同时举手 → 货交接成功

---

### 3.3 SOF (帧起始) 与 EOL (行结束)

```
一帧图像的结构：

SOF ─▶ [第0行像素...像素] EOL
        [第1行像素...像素] EOL
        [第2行像素...像素] EOL
        ...
        [第1079行像素...像素] EOL
    
SOF 标志新一帧开始，EOL 标志一行结束。

FPGA 通过这两个信号知道：
- SOF 时 → 把坐标重置为 (0, 0)
- EOL 时 → 行号+1，列号归零
```

## 四、坏点校正核心概念

### 4.1 坏点表 (Bad Pixel Table)

一张"黑名单"，记录所有坏点的坐标位置。

```
坏点表示例（预先存储在硬件里）：
条目0: (X=100, Y=50)   ← 第50行、第100列的像素是坏点
条目1: (X=320, Y=128)
条目2: (X=1024, Y=200)
...
最多 1024 条
```

---

### 4.2 坐标跟踪器

一个**内置计数器**，实时追踪"当前正在处理第几行、第几列"。

```
当前是第 Y 行、第 X 列的像素 → 查坏点表：这个(X,Y)在不在表里？
```

---

### 4.3 行缓冲 (Line Buffer)

一个**延时存储器**，把几行前的像素暂存起来，供后续使用。

**为什么需要它？**

FPGA 的视频流是单向的——像素像流水一样经过，只能看到"当前"和"过去"的像素，看不到"未来"的。但要校正当前坏点，需要用到上方像素（已经过去了），所以要把过去的像素缓存下来。

```
时间线：
  [过去] 像素已经流过 → 存在 Line Buffer 里
  [现在] 当前像素正在处理
  [未来] 像素还没到 → 无法使用

Line Buffer = "时光机"，让你能回头看已经过去的像素
```

---

### 4.4 二分查找

在已排序列表里高效查找的方法。

```
普通查找（逐条比较）：最坏要比较 1024 次
二分查找：每次砍掉一半 → 最多只要比较 10 次

举例：在 1~1024 之间猜一个数
- 猜 512，"大了" → 范围缩小至 1~511
- 猜 256，"小了" → 范围缩小至 257~511
- 猜 384，"对了！" → 3 步找到
```

**在 FPGA 中的实现**：把 10 步比较做成 10 级流水线，每级处理不同的像素。这样虽然单个像素要等 10 拍才有结果，但每个时钟仍有 1 个新结果出来。

---

### 4.5 MUX (多路选择器)

一个"二选一开关"：

```
         ┌─── 原值（不是坏点，直通）
         │
输入 ────┤
         │
         └─── 校正值（是坏点，替换）

控制信号：is_bad_pixel ? 选校正值 : 选原值
```

---

### 4.6 BYPASS 模式（旁路模式）

"**直通模式**"——输入数据不做任何处理，原封不动地输出。

**为什么需要？**
- 调试时先跑 BYPASS 确认整个数据通路是通的
- 对比"处理前 vs 处理后"的效果
- 出问题时可以随时关掉校正功能

---

## 五、寄存器接口

### 5.1 什么是寄存器

FPGA 内部的一组"控制面板旋钮"。上位机（比如 CPU 或电脑）可以通过读写寄存器来控制 IP 的行为、查询当前状态。

**比喻**：寄存器就像家电上的按钮和显示屏——你按"开机"按钮（写控制寄存器），灯亮了（读状态寄存器）。

---

### 5.2 寄存器访问属性

| 属性 | 全称 | 含义 |
|------|------|------|
| RO | Read Only | 只读——只能看，不能改（如版本号） |
| RW | Read/Write | 可读可写——可以改配置（如使能开关） |
| WO | Write Only | 只写——只能发命令，读回来没意义（如复位） |

---

### 5.3 本项目关键寄存器速查

| 寄存器 | 通俗功能 |
|--------|----------|
| IP_CTRL | **总开关**：使能校正、BYPASS 旁路、调试模式 |
| IMG_WIDTH / IMG_HEIGHT | 告诉 IP 当前图像是 1920×1080 还是 3840×2160 |
| IMG_FORMAT | 告诉 IP 每个像素是 8-bit、10-bit 还是 12-bit |
| BAD_TABLE_* | 向上位机提供坏点表的写入通道 |
| BAD_HIT_COUNT | 统计到目前为止命中了多少次坏点（调试用） |

## 六、开发流程术语

### 6.1 RTL (Register Transfer Level)

**寄存器传输级**——用 Verilog 代码描述硬件电路行为的设计层次。

**通俗理解**：RTL 代码 = 硬件电路的设计图纸，写 RTL = 画电路图（只不过用代码画）。

---

### 6.2 仿真 (Simulation) / Testbench (测试平台)

在电脑上用软件模拟 FPGA 电路的行为，看它处理数据对不对。

- **仿真** = 虚拟运行电路
- **Testbench** = 给电路输入模拟数据并检查输出的"测试程序"
- **波形 (Waveform)** = 仿真结果的可视化展示，像示波器一样看每个信号的变化

**为什么必须先仿真？** 因为直接烧到 FPGA 板子上如果出错，很难定位。仿真可以精确看到每个时钟每个信号的值。

---

### 6.3 综合 (Synthesis)

把 Verilog 代码"翻译"成 FPGA 能理解的逻辑单元（LUT、FF、BRAM 的连接关系）。

```
Verilog 代码 ──[综合]──▶ 门级网表（LUT+FF+连线的列表）
```

**比喻**：综合 = 把建筑设计图翻译成砖头、水泥、钢筋的具体用量清单。

---

### 6.4 网表 (Netlist)

综合后的输出——一个列举了所有逻辑单元和它们之间连线关系的列表。

---

### 6.5 实现 (Implementation) / 布局布线 (Place & Route)

把网表中的逻辑单元分配到 FPGA 芯片的具体物理位置，并连好线。

```
网表 ──[布局布线]──▶ 物理电路
```

**比喻**：实现 = 在 FPGA 这个"小区"里，给每个逻辑单元分配具体"房间号"，并铺设连接它们的"走廊"。

---

### 6.6 比特流 (Bitstream)

最终烧录到 FPGA 里的二进制文件（.bit 文件）。

它告诉 FPGA 内部每一个开关是开还是关，形成目标电路。

---

### 6.7 板级验证 (Board-Level Verification)

把 bit 文件下载到真实的 awesom 核心板上，接上真实的视频信号（或测试图案），用眼睛或仪器确认效果。

| 仿真   | 板级验证          |
| ---- | ------------- |
| 软件模拟 | 真实硬件          |
| 看波形   | 看实际画面         |
| 快速   | 慢（要烧录、接线）     |
| 不能完全代替 | 最终确认——板上跑过才算过 |

---

### 6.8 覆盖率 (Coverage)

仿真测试"覆盖"了多少代码的百分比。比如：
- **行覆盖率**：多少行代码被执行过
- **分支覆盖率**：多少 if/else 分支都被测试到
- **条件覆盖率**：每个条件中的每个子条件都测试过

**目标**：覆盖率越高 → 测试越充分 → bug 越少。

---

### 6.9 时序约束 (Timing Constraints)

告诉综合工具时间上的要求，比如"这个信号必须在 8ns 内稳定"。

三种常见约束：
```
create_clock    → 告诉工具时钟频率是多少
set_input_delay  → 输入信号到达芯片的时间要求
set_output_delay → 输出信号到达下一级的时间要求
```

## 八、一句话总结

```
坏点校正的本质：

"摄像头传感器上有一些坏掉的像素，
我们用一个硬件电路（FPGA），
在视频流经过时，
```
